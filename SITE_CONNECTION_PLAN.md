# Create Studio Registration & Site Connection Flow - Implementation Plan

## Problem Statement

Create Studio needs a unified registration and site connection system that supports:
1. **Web Registration** → Users register on create.studio → Add sites from dashboard → Connect via plugin
2. **Plugin Registration** → Users directed to web registration → Return to plugin to connect
3. **Existing Users** → V1 users with no password can upgrade → Manage sites on web

## Solution: Plugin-Generated Site Verification

**Approach**: The WordPress plugin generates a site-specific **site verification code**. Users copy this code from the plugin settings and paste it on Create Studio to verify their access. Create Studio verifies the code by calling the plugin's REST endpoint directly.

**Key Security Principle**: The site verification code originates from the WordPress plugin (not from Create Studio via email). This ensures that anyone who provides the code has actual WP admin access to view the plugin settings page.

**Terminology**:
- **Site verification code**: The 32-character code generated by the WordPress plugin, used to prove WP admin access
- **Verified**: A SiteUsers record where `verified_at` is set (user has proven access to the site)
- **Pending**: A SiteUsers record where `verified_at` is NULL (awaiting verification)

**Why Plugin-Generated Codes?**
- ✅ Secure (code can only be obtained by WP admin users)
- ✅ No email interception vulnerability
- ✅ Create Studio verifies directly with plugin (can't be spoofed)
- ✅ Verifies individual user ownership (requires viewing plugin settings)
- ✅ Simple UX (copy/paste code)
- ✅ No OAuth/webhooks needed
- ✅ Same code works for multiple users of same site (team-friendly)
- ✅ **Prevents privilege escalation** (each user must verify independently)

## Implementation Plan

### Phase 1: Database Schema (Priority 1)

**Critical Change**: Connection tracking happens on the **SiteUsers table**, not on Sites or a separate table. This ensures each user-site relationship has its own independent verification state.

Use `npx nuxthub database migrations create <name>` to create a new migration file.
Example: `npx nuxthub database migrations create add-site-user-verification`

```sql
-- Add verification tracking to SiteUsers table
ALTER TABLE SiteUsers ADD COLUMN verified_at DATETIME;

-- Mark ALL existing SiteUsers as verified (they were created via V1 plugin registration)
UPDATE SiteUsers SET verified_at = CURRENT_TIMESTAMP WHERE verified_at IS NULL;

CREATE INDEX idx_site_users_verified_at ON SiteUsers(verified_at);
```

**Migration Note**: The `UPDATE` statement marks all existing SiteUsers records as verified because they were created either:
1. Through V1 plugin registration (user proved ownership via that site's plugin)
2. Through the `migrate-canonical-sites` script (which consolidates V1 user-site relationships)

Both cases represent users who already verified their site access through the plugin.

**How It Works**:
1. User adds a site → Creates row in `SiteUsers` with `verified_at = NULL` (pending)
2. User copies site verification code from WordPress plugin settings (proves WP admin access)
3. User pastes code on Create Studio → Create Studio verifies with plugin REST endpoint
4. Plugin confirms code → Create Studio sets `SiteUsers.verified_at = NOW()` (verified)
5. Each user-site pair tracks its own verification state independently
6. Even if User A has verified access to site.com, User B must verify independently

**Column Meanings**:
- `verified_at`: When this user verified their access to this site (NULL = pending, set = verified)

**Note**: Site verification codes are generated and stored by the WordPress plugin, not Create Studio. This is more secure because:
- Code can only be obtained by someone with WP admin access
- Create Studio verifies directly with the plugin (can't be spoofed)
- No email interception vulnerability

**This prevents**:
- User A adds site.com and verifies, User B can't just add site.com and get access
- User B must copy the site verification code from plugin settings and verify themselves
- Each user independently verifies their access to each site
- Per-user verification is baked into the core relationship model

### Phase 1.5: Plugin-Generated Site Verification Code (Priority 1)

**Key Insight**: Email-based verification codes are vulnerable to interception. A malicious actor who receives the email (e.g., through email forwarding or compromise) could verify connection to any site. Instead, the code should originate from the WordPress plugin itself, proving the user has WP admin access.

**Security Problem with Email-Based Codes**:
- User registers on Create Studio
- Create Studio emails a verification code
- Attacker intercepts email (forwarding, compromise, etc.)
- Attacker can now verify connection to ANY site by fabricating the `site_url` parameter
- The code proves nothing about site ownership

**New Approach - Plugin-Generated Site Verification Code**:
1. WordPress plugin generates a unique site verification code (on first install or on-demand)
2. Code is stored permanently in `mv_settings` table (reusable for all users of that site)
3. User copies code from WP admin plugin settings page (proves WP admin access)
4. User pastes code on Create Studio verification form
5. Create Studio calls plugin REST endpoint to verify code matches
6. If code matches → user's SiteUsers record is marked as verified

**Why This is Secure**:
- Site verification code can only be obtained by someone with WP admin access to view plugin settings
- No email interception vector
- Create Studio verifies with the plugin directly (can't fake the verification)
- Same code works for all users of that site (simple, no per-user complexity on plugin side)
- Per-user tracking still happens in SiteUsers table on Create Studio side

**WordPress Plugin Changes**:

1. **Generate Site Verification Code** (on-demand when user clicks button):
```php
// Get the current site verification code (or null if not yet generated)
function mv_create_get_site_verification_code() {
    global $wpdb;
    $table = $wpdb->prefix . 'mv_settings';
    return $wpdb->get_var(
        $wpdb->prepare(
            "SELECT setting_value FROM $table WHERE setting_key = %s",
            'site_verification_code'
        )
    );
}

// Generate a new code - called when user clicks "Generate Code" button
function mv_create_generate_verification_code() {
    global $wpdb;
    $table = $wpdb->prefix . 'mv_settings';

    // Generate new code (32-char alphanumeric)
    $code = wp_generate_password(32, false, false);

    // Check if code already exists
    $existing = $wpdb->get_var(
        $wpdb->prepare(
            "SELECT id FROM $table WHERE setting_key = %s",
            'site_verification_code'
        )
    );

    if ($existing) {
        // Update existing code
        $wpdb->update(
            $table,
            [
                'setting_value' => $code,
                'updated_at' => current_time('mysql'),
            ],
            ['setting_key' => 'site_verification_code']
        );
    } else {
        // Insert new code
        $wpdb->insert(
            $table,
            [
                'setting_key' => 'site_verification_code',
                'setting_value' => $code,
                'created_at' => current_time('mysql'),
            ]
        );
    }

    return $code;
}

// AJAX endpoint for generating code (called from plugin settings page)
add_action('wp_ajax_mv_create_generate_code', function() {
    // Security: Only allow logged-in admins
    if (!current_user_can('manage_options')) {
        wp_send_json_error(['message' => 'Insufficient permissions'], 403);
    }

    $code = mv_create_generate_verification_code();
    wp_send_json_success(['code' => $code]);
});
```

3. **REST Endpoint to Verify Code** (public, called by Create Studio):
```php
// Register REST endpoint
add_action('rest_api_init', function() {
    register_rest_route('mv-create/v1', '/verify-site-code', [
        'methods' => 'POST',
        'callback' => 'verify_site_verification_code',
        'permission_callback' => '__return_true', // Public endpoint
    ]);
});

function verify_site_verification_code($request) {
    global $wpdb;

    $submitted_code = $request->get_param('code');
    $table = $wpdb->prefix . 'mv_settings';
    $stored_code = $wpdb->get_var(
        $wpdb->prepare(
            "SELECT setting_value FROM $table WHERE setting_key = %s",
            'site_verification_code'
        )
    );

    if (!$stored_code) {
        return new WP_REST_Response(['valid' => false, 'error' => 'Site verification code not configured'], 400);
    }

    if (!hash_equals($stored_code, $submitted_code)) {
        return new WP_REST_Response(['valid' => false, 'error' => 'Invalid verification code'], 401);
    }

    return new WP_REST_Response([
        'valid' => true,
        'site_url' => home_url(),
        'wp_version' => get_bloginfo('version'),
        'php_version' => phpversion(),
        'create_version' => MV_CREATE_VERSION,
    ], 200);
}
```

**Create Studio Verification Flow**:

1. User adds site URL on Create Studio dashboard
2. Create Studio creates pending `SiteUsers` record (no code stored here - code lives on plugin only)
3. User copies site verification code from WordPress plugin settings
4. User pastes code on Create Studio verification form
5. Create Studio calls `POST {site_url}/wp-json/mv-create/v1/verify-site-code` with `{ code: "..." }`
6. If plugin returns `{ valid: true }`:
   - Mark `SiteUsers.verified_at = NOW()` (pending → verified)
   - Mark `Users.validEmail = 1` (user has working site access = valid email assumption)
   - Return success to user

- Site verification codes are stored ONLY on the WordPress plugin side (`mv_settings` table in the plugin database)
- Create Studio only tracks `SiteUsers.verified_at` to know if user has been verified
- No code columns needed on Create Studio database

#### 2.2 URL Validation and Normalization Utility

**File**: `packages/app/server/utils/url.ts` (NEW)

```typescript
/**
 * Normalizes and validates a WordPress site URL for storage and comparison.
 * Returns null if the URL is invalid or not allowed.
 */
export function normalizeSiteUrl(input: string): string | null {
  try {
    // Parse the URL
    let url = new URL(input.trim())

    // Enforce HTTPS (required for secure plugin communication)
    if (url.protocol !== 'https:') {
      // Allow http for localhost during development only
      if (url.hostname !== 'localhost' && !url.hostname.endsWith('.local')) {
        return null // Reject non-HTTPS URLs
      }
    }

    // Block private/internal IP ranges (security)
    const hostname = url.hostname
    if (isPrivateOrReservedHost(hostname)) {
      return null
    }

    // Normalize: lowercase hostname
    url.hostname = url.hostname.toLowerCase()

    // Normalize: remove default ports
    if ((url.protocol === 'https:' && url.port === '443') ||
        (url.protocol === 'http:' && url.port === '80')) {
      url.port = ''
    }

    // Normalize: remove trailing slash from path
    url.pathname = url.pathname.replace(/\/+$/, '') || '/'

    // Normalize: remove hash and search params (not part of site identity)
    url.hash = ''
    url.search = ''

    // Return normalized URL without trailing slash
    let normalized = url.origin + (url.pathname === '/' ? '' : url.pathname)

    return normalized
  } catch {
    return null // Invalid URL
  }
}

/**
 * Checks if a hostname is private, reserved, or potentially malicious.
 */
function isPrivateOrReservedHost(hostname: string): boolean {
  // Block localhost variants
  if (hostname === 'localhost' || hostname === '127.0.0.1' || hostname === '::1') {
    // Allow localhost only in development
    return process.env.NODE_ENV === 'production'
  }

  // Block private IP ranges
  const ipv4Private = /^(10\.|172\.(1[6-9]|2[0-9]|3[0-1])\.|192\.168\.|169\.254\.)/
  if (ipv4Private.test(hostname)) {
    return true
  }

  // Block link-local and reserved
  if (hostname.endsWith('.local') || hostname.endsWith('.internal')) {
    return process.env.NODE_ENV === 'production'
  }

  return false
}

/**
 * Validates that a URL looks like a WordPress site.
 * Does not make network requests - just validates format.
 */
export function isValidWordPressSiteUrl(url: string): boolean {
  const normalized = normalizeSiteUrl(url)
  if (!normalized) return false

  // Must have a valid TLD (or be localhost in dev)
  const parsed = new URL(normalized)
  const hostname = parsed.hostname

  // Basic TLD check (has at least one dot, or is localhost)
  if (!hostname.includes('.') && hostname !== 'localhost') {
    return false
  }

  return true
}
```

**File**: `packages/app/server/api/v2/sites/[siteId]/verify.post.ts` (NEW)

**Endpoint**: `POST /api/v2/sites/:siteId/verify`
- Requires: Session authentication
- Body: `{ verification_code: string }`
- **Key Logic**:
  1. Find site by ID
  2. Find SiteUsers record for current user + this site
  3. If already verified → return error
  4. Call WordPress plugin to verify code (with timeout and error handling):
     ```typescript
     try {
       const pluginResponse = await $fetch(
         `${site.url}/wp-json/mv-create/v1/verify-site-code`,
         {
           method: 'POST',
           body: { code: verification_code },
           timeout: 10000, // 10 second timeout
           retry: 1, // Retry once on transient failures
         }
       )

       if (pluginResponse.valid) {
         // Mark SiteUsers.verified_at = NOW()
         await siteUserRepo.markVerified(siteUser.id)
         // Mark Users.validEmail = 1 (user has working site = valid email assumption)
         await userRepo.setValidEmail(user.id)
         // Update site metadata from plugin response
         await siteRepo.updateMetadata(site.id, {
           wp_version: pluginResponse.wp_version,
           php_version: pluginResponse.php_version,
           create_version: pluginResponse.create_version,
         })
         return { success: true, site, verified_at: new Date().toISOString() }
       } else {
         return { success: false, error: pluginResponse.error || 'Invalid verification code' }
       }
     } catch (error: any) {
       // Handle network/timeout errors gracefully
       if (error.code === 'ETIMEDOUT' || error.code === 'ECONNREFUSED' || error.code === 'ENOTFOUND') {
         throw createError({
           statusCode: 502,
           message: 'Could not reach your WordPress site. Please ensure the site is accessible and the Create plugin is active.'
         })
       }
       if (error.statusCode === 404) {
         throw createError({
           statusCode: 502,
           message: 'Create plugin REST endpoint not found. Please ensure the Create plugin is installed and activated.'
         })
       }
       // Re-throw unexpected errors
       throw error
     }
     ```
- Returns: `{ success: true, site, verified_at }` or `{ success: false, error: "..." }`

**Security Notes**:
- Create Studio is the client calling the plugin, not vice versa
- Plugin verifies the code matches what it generated
- No way for attacker to spoof verification (would need to control plugin endpoint)
- Rate limiting recommended: 5 attempts per user per hour per site
- 10-second timeout prevents hanging on unresponsive sites
- Single retry handles transient network issues

#### 6.3 Site Verification Code Generation and Storage

The plugin generates and stores the site verification code in the `mv_settings` table **on-demand** when the user clicks the "Generate Code" button.

**PHP Implementation**:

```php
// Get the current site verification code (or null if not generated yet)
function mv_create_get_site_verification_code() {
    global $wpdb;
    $table = $wpdb->prefix . 'mv_settings';
    return $wpdb->get_var(
        $wpdb->prepare(
            "SELECT setting_value FROM $table WHERE setting_key = %s",
            'site_verification_code'
        )
    );
}

// Generate a new code - called when user clicks "Generate Code" button
function mv_create_generate_verification_code() {
    global $wpdb;
    $table = $wpdb->prefix . 'mv_settings';

    // Generate new code (32-char alphanumeric)
    $code = wp_generate_password(32, false, false);

    // Check if code already exists
    $existing = $wpdb->get_var(
        $wpdb->prepare(
            "SELECT id FROM $table WHERE setting_key = %s",
            'site_verification_code'
        )
    );

    if ($existing) {
        // Update existing code
        $wpdb->update(
            $table,
            [
                'setting_value' => $code,
                'updated_at' => current_time('mysql'),
            ],
            ['setting_key' => 'site_verification_code']
        );
    } else {
        // Insert new code
        $wpdb->insert(
            $table,
            [
                'setting_key' => 'site_verification_code',
                'setting_value' => $code,
                'created_at' => current_time('mysql'),
            ]
        );
    }

    return $code;
}

// REST endpoint for generating code (called from plugin settings page via AJAX)
add_action('wp_ajax_mv_create_generate_code', function() {
    // Security: Only allow logged-in admins
    if (!current_user_can('manage_options')) {
        wp_send_json_error(['message' => 'Insufficient permissions'], 403);
    }

    $code = mv_create_generate_verification_code();
    wp_send_json_success(['code' => $code]);
});
```

**JavaScript/React Handler** (in `APIRegistrationPrompt.tsx` or similar):

```javascript
async function handleGenerateCode() {
    setIsGenerating(true);
    try {
        const response = await fetch(ajaxurl, {
            method: 'POST',
            headers: {
                'Content-Type': 'application/x-www-form-urlencoded',
            },
            body: new URLSearchParams({
                action: 'mv_create_generate_code',
                nonce: mvCreateNonce, // WordPress nonce for security
            }),
        });

        const data = await response.json();

        if (data.success) {
            setGeneratedCode(data.data.code);
            setShowCode(true);
        } else {
            setError(data.data.message || 'Failed to generate code');
        }
    } catch (error) {
        setError('Error generating code: ' + error.message);
    } finally {
        setIsGenerating(false);
    }
}

function handleCopyCode() {
    navigator.clipboard.writeText(generatedCode).then(() => {
        setCopied(true);
        setTimeout(() => setCopied(false), 2000); // Reset after 2 seconds
    });
}
```

## Security Considerations

✅ **Plugin-Generated Code Model** (CRITICAL)
- Connection code is generated by the WordPress plugin, not Create Studio
- Code can only be obtained by someone with WP admin access (view plugin settings page)
- Create Studio verifies code by calling plugin REST endpoint directly (can't be spoofed)
- Eliminates email interception vulnerability from previous approach

✅ **Per-User Verification Model** (CRITICAL)
- Verification tracking is baked into SiteUsers table - each user-site pair is independent
- Even if User A verifies site.com, User B cannot gain access without their own verification
- User B must copy the site verification code from plugin settings and verify independently on Create Studio
- `verified_at` field ensures user must complete verification before access granted
- Prevents privilege escalation and unauthorized access

✅ **Site Verification Code Security**
- 32-character alphanumeric (high entropy)
- Generated once per site, stored in WordPress plugin `mv_settings` table
- Same code reusable by multiple users (proves access to same site)
- Code can be regenerated by site admin if compromised

✅ **Site Hijacking Prevention**
- Site verification code only valid for the site that generated it (plugin returns site URL in response)
- Create Studio verifies the site URL matches the claimed site
- Each user must independently prove admin access by viewing plugin settings
- No way to fake verification without controlling the WordPress site

✅ **Privilege Escalation Prevention**
- User pending (unverified) until `verified_at IS NOT NULL`
- Pending records in SiteUsers grant NO access to site data
- Plugin verification is required step
- Access only granted after Create Studio successfully verifies with plugin

✅ **Rate Limiting**
- 5 attempts per hour per user per site on verification endpoint
- Prevents brute force attacks on site verification codes
- Plugin endpoint is public but code verification is timing-safe

✅ **Subscription Protection**
- Only verified (verified_at NOT NULL) users can manage subscriptions
- Check user exists in `SiteUsers` with `verified_at` set
- Prevents billing for unverified users

✅ **V1 Compatibility**
- Existing V1 sites remain connected (migration sets `verified_at`)
- V1 users who want to add NEW sites must go through web registration flow
- JWT tokens are site-specific (can't be used to add different sites)
- No disruption to current users on existing sites
